## Behavior

- If the tool being called is **not** a Jira write → exit silently
  (allow).
- If the tool **is** a Jira write **and** `DRY_RUN=true` → allow
  silently. The jira MCP server itself will short-circuit the write to a
  stderr log, so no real Jira mutation happens.
- If the tool **is** a Jira write **and** `DRY_RUN` is unset → emit a
  `permissionDecision: "ask"` with a clear explanation. The user has to
  explicitly confirm in the Claude Code UI before the write proceeds.

This is the production pattern for any high-blast-radius MCP tool: keep
the safety check at the hook layer, **not just** at the server layer,
because the hook fires BEFORE the tool call leaves Claude's context —
giving the user a chance to stop it.

## The gated tools

```js
const WRITE_TOOLS = new Set([
  "mcp__jira__jira_create_issue",
  "mcp__jira__jira_add_comment",
]);
```

Two tools, both irreversibly mutate state in someone else's system
(your Jira instance). Reads (`jira_search_issues`, `jira_get_issue`,
`jira_list_projects`) are not gated — they're idempotent.

## The confirmation prompt

When triggered, the user sees:

```text
⚠ Jira write requested: create Jira issue in project FLEXPAY: "Eligibility migration capacity ask"

DRY_RUN is not set, so this would hit the real Jira instance. Confirm
before proceeding.

To always allow Jira writes during this session, retry with DRY_RUN
unset and accept here.

To never write to real Jira, set DRY_RUN=true in your environment.
```

The prompt summarizes the actual mutation (project key + summary for
create, issue key + body excerpt for comment). That's what the user is
authorizing — not "Jira write" in the abstract, but this specific
ticket.

## The full hook

```js
#!/usr/bin/env node
const WRITE_TOOLS = new Set([
  "mcp__jira__jira_create_issue",
  "mcp__jira__jira_add_comment",
]);

let raw = "";
process.stdin.setEncoding("utf8");
process.stdin.on("data", (chunk) => (raw += chunk));
process.stdin.on("end", () => {
  let event;
  try {
    event = JSON.parse(raw);
  } catch {
    // Malformed input — fail open (don't block the tool call on hook bugs).
    process.exit(0);
  }

  const toolName = event?.tool_name ?? "";
  if (!WRITE_TOOLS.has(toolName)) {
    process.exit(0);
  }

  if (process.env.DRY_RUN === "true") {
    // DRY_RUN handles the safety at the MCP server layer; nothing to gate.
    process.exit(0);
  }

  const input = event?.tool_input ?? {};
  const summary =
    toolName === "mcp__jira__jira_create_issue"
      ? `create Jira issue in project ${input.project_key}: "${input.summary}"`
      : `add comment to ${input.key}: "${(input.body || "").slice(0, 120)}${(input.body || "").length > 120 ? "…" : ""}"`;

  const reason = [
    `⚠ Jira write requested: ${summary}`,
    ``,
    `DRY_RUN is not set, so this would hit the real Jira instance. Confirm before proceeding.`,
    `To always allow Jira writes during this session, retry with DRY_RUN unset and accept here.`,
    `To never write to real Jira, set DRY_RUN=true in your environment.`,
  ].join("\n");

  console.log(
    JSON.stringify({
      hookSpecificOutput: {
        hookEventName: "PreToolUse",
        permissionDecision: "ask",
        permissionDecisionReason: reason,
      },
    }),
  );
  process.exit(0);
});
```

## Failure-open semantics

If the event JSON is malformed or the hook crashes, the tool call
proceeds. A buggy hook shouldn't take down the agent.

This is fine here because the **MCP server** also enforces `DRY_RUN` —
the hook is a first-line UX-layer gate, the server is the second-line
network-layer gate. Both have to be defeated for an unintended mutation
to land. The hook is *not* a sole line of defense; it's defense in
depth.

<Aside type="tip">
The same shape generalizes to any high-blast-radius MCP tool. If you
add a server that wraps Linear / GitHub / Confluence / an internal API
with write tools, add an entry to `WRITE_TOOLS` and the same hook
covers it.
</Aside>

## Tests

`tests/hooks.test.ts` runs the hook against a small set of synthetic
events:

- Non-Jira tool → exit 0, no output.
- Jira write tool + `DRY_RUN=true` → exit 0, no output.
- Jira write tool + `DRY_RUN` unset → exit 0, `permissionDecision: "ask"`
  with the expected reason shape.
- Malformed JSON → exit 0, fail-open.
