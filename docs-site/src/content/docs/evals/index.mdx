## Run it

```bash
python evals/run.py                 # all skills
python evals/run.py meeting-killer  # one skill
python evals/run.py oncall-companion
```

Per-case outputs are captured to `evals/runs/<ts>/` so you can read what
the agent actually produced when a judge verdict is unexpected. The
top-level report is at `evals/report.md`.

## How it works

1. **Load cases.** Each `evals/golden/<skill>.yaml` declares a set of
   cases with `id`, `description`, an input (question or transcript path),
   `must_have` properties, and `must_not_have` properties.
2. **Run the agent.** The harness invokes the same agent the user would
   — same model, same prompts, same MCP servers (synthetic mode in CI).
3. **Judge.** The agent's full output is handed to a separate judge
   model call (Opus 4.7) along with the rubric. The judge scores each
   `must_have` and `must_not_have` independently and produces an overall
   pass/fail.
4. **Report.** Markdown report written to `evals/report.md` with
   per-case verdict, judge reasoning, and token cost.

## Why this matters

Most "AI for engineering" deployments don't have eval coverage at all.
The Pragmatic Engineer 2026 survey of 906 engineers doesn't even mention
evals.

The eval-gate is the difference between "behavior changes randomly with
every prompt edit" and "behavior changes only when you intend it to." For
production AI tools at a regulated company, this is the difference
between maintainable and unmaintainable.

<Aside type="tip">
The judge model is the same as the agent model (Opus 4.7). That's not
a bug — the judge gets a different prompt with explicit grading
instructions and the rubric, so the asymmetry between "produce an answer
to this question" and "grade whether the answer matches the rubric" is
strong enough to keep them honest. If you want a stronger separation,
swap the judge to a different vendor's model and the runner abstraction
holds.
</Aside>
