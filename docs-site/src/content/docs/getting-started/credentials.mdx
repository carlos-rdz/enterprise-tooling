Out of the box, the `slack` and `jira` MCP servers return synthetic
FlexPay-shaped responses. That's enough for every demo and every test in
this repo. When you want to point them at your own workspace:

<Steps>

1. **Copy the env template.**

   ```bash
   cp .env.example .env
   ```

2. **Fill in your tokens** (described per-provider below). Leave
   `DRY_RUN=true` while you're getting comfortable — writes log to stderr
   instead of executing.

3. **Restart any running MCP servers** so they re-read env vars on startup.

</Steps>

<Aside type="caution">
`.env` is gitignored. Never commit tokens. The
[secret scanner hook](/enterprise-tooling/hooks/secret-scanner/) blocks any
write payload containing known credential prefixes — but the first line of
defense is keeping the secrets out of your repo entirely.
</Aside>

## Slack

<Steps>

1. **Create a Slack app** at https://api.slack.com/apps → "Create New App"
   → "From scratch".

2. **Add bot scopes** under OAuth & Permissions:

   - `channels:history`
   - `channels:read`
   - `groups:history`
   - `groups:read`
   - `im:write`
   - `chat:write`
   - `search:read`
   - `users:read`
   - `users:read.email`

3. **Install the app** to your workspace and copy the **Bot User OAuth
   Token** (starts with `xoxb-`).

4. **Set it in `.env`:**

   ```bash
   SLACK_BOT_TOKEN=xoxb-...
   ```

</Steps>

The `slack` MCP server detects the token on startup and switches from
synthetic to live mode. Write operations (`slack_post_message`, `slack_send_dm`)
still honor `DRY_RUN=true` — they log the would-have-sent payload to stderr
instead of hitting the Slack API.

## Jira Cloud

<Steps>

1. **Create an API token** at
   https://id.atlassian.com/manage-profile/security/api-tokens.

2. **Set the values in `.env`:**

   ```bash
   JIRA_HOST=yourco.atlassian.net   # your Atlassian subdomain, no protocol
   JIRA_EMAIL=you@yourco.com         # email tied to the API token
   JIRA_API_TOKEN=...
   ```

3. **Verify** by calling `health_check` on the jira MCP server — it will
   hit `/myself` and return the authenticated user.

</Steps>

<Aside type="note">
The `jira` MCP server uses basic auth (email + token). Scope the bot
account, not the token. The token inherits whatever the account can do.
</Aside>

## DRY_RUN

```bash
DRY_RUN=true
```

When set, every write operation on the live MCP servers (Slack post/DM,
Jira create/comment) short-circuits to a stderr log instead of executing.
This is the default in `.env.example` and the recommended posture for
demoing against a real workspace.

The Claude Code [Jira write gate hook](/enterprise-tooling/hooks/jira-write-gate/)
also fires before any Jira write tool — when `DRY_RUN` is unset it asks for
explicit user confirmation in the UI. Two layers, intentional.

## Operational mode summary

| Setting | Slack | Jira | Behavior |
|---|---|---|---|
| Nothing set | synthetic | synthetic | Fixtures only. Safe for cloning + demos. |
| Tokens set, `DRY_RUN=true` | live reads, dry writes | live reads, dry writes | Real data in, no mutations. The default. |
| Tokens set, `DRY_RUN` unset | live | live | Full read+write. The Jira-write hook gates mutations at the UI. |

See [Three operational modes](/enterprise-tooling/concepts/three-modes/) for
why these specific three exist.
