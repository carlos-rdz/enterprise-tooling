## What it does

You almost always don't know what you don't know. That's why this workflow
exists. It triangulates three signals you would otherwise have to
hand-merge:

<CardGrid>
  <Card title="team-registry" icon="seti:graphql">
    Per-team mission, lead, current sprint, and recent decisions. The
    organizational ground truth.
  </Card>
  <Card title="Jira (live)" icon="seti:jenkins">
    Active and recently-moved tickets across teams. The work-in-flight
    ground truth.
  </Card>
  <Card title="Slack (live)" icon="seti:slim">
    Recent messages in cross-functional channels. The "what people are
    actually talking about" ground truth.
  </Card>
</CardGrid>

Output: the 2-3 critical hidden dependencies or overlaps, concrete owners
to talk to today (by name), and the single biggest "align on this week or
ship slips a quarter" warning.

## The two implementations

<Tabs>
<TabItem label="Raw Python form">

`03_cross_team/agent.py` — uses a **manual tool-use loop**. The agent
keeps calling tools until it produces an answer with no more
`tool_use` blocks. This is the rawest form of an agentic loop in the
Anthropic SDK.

```py
tools = [list_teams_tool, get_team_activity_tool, search_across_teams_tool,
         jira_search_tool, slack_search_tool]

messages = [{"role": "user", "content": question}]

while True:
    resp = client.messages.create(
        model="claude-opus-4-7",
        max_tokens=8000,
        thinking={"type": "enabled", "budget_tokens": 4000},
        tools=tools,
        messages=messages,
    )
    messages.append({"role": "assistant", "content": resp.content})
    if resp.stop_reason != "tool_use":
        break
    tool_results = [execute(block) for block in resp.content if block.type == "tool_use"]
    messages.append({"role": "user", "content": tool_results})
```

The agent picks its own search terms and pivots based on what it finds —
no hardcoded query plan. OpenTelemetry instrumentation wraps the loop so
you can see per-tool latency in your tracing backend (see
[Observability](/enterprise-tooling/guides/observability/)).

Run it:

```bash
python 03_cross_team/agent.py "who else is touching biometric auth right now"
```

```md
# Cross-Team Integrator

You proactively discover what OTHER teams are doing that overlaps with the
user's work.

## How to answer
1. Start broad: `list_teams` to know what teams exist.
2. Search for the concepts in the user's question across ALL teams.
3. When you find a match in another team, fetch that team's full activity.
4. Look for second-order overlaps.

## Output structure
1. The 2-3 critical hidden dependencies or overlaps
2. Concrete owners to talk to TODAY — by name
3. The single biggest "align on this week or ship slips a quarter" warning
```

Invoke from Claude Code:

```
/cross-team I'm leading the FlexPay Q3 Standard expansion. What should I be worried about across teams?
```

## Eval coverage

`evals/golden/cross-team.yaml` — 2 cases. The judge specifically checks
that the agent:

- Names **specific people** as the action owners, not just teams.
- Identifies the UDAAP gate as the schedule-defining constraint.
- Catches the second-order overlap on the deprecated biometric library
  (PLAN-1925 + MOBILE-3501 + AUTH-455 + FRAUD-1120 stacking on a lib
  that's about to be deleted).

That last one is the whole point of the workflow — it's the kind of thing
that's impossible for any one person on any one team to see from their
seat.
