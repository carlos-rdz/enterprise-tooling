**Source:** `.claude/hooks/scan-secrets.mjs`

Fires before any of:

```
mcp__slack__slack_post_message
mcp__slack__slack_send_dm
mcp__jira__jira_create_issue
mcp__jira__jira_add_comment
```

Walks every string in `tool_input` and checks it against three layers
of detection. On a positive match, emits
`permissionDecision: "deny"` with a clear reason. The agent sees the
denial and is prompted to remove the secret before retrying.

## The 14 credential patterns

| Label | Pattern |
|---|---|
| Slack bot token | `\bxoxb-[A-Za-z0-9-]{20,}` |
| Slack user token | `\bxoxp-[A-Za-z0-9-]{20,}` |
| Slack app token | `\bxapp-[A-Za-z0-9-]{20,}` |
| Slack webhook URL | `https://hooks\.slack\.com/services/[A-Z0-9]+/[A-Z0-9]+/[A-Za-z0-9]+` |
| GitHub PAT | `\bgh[poursu]_[A-Za-z0-9]{36,}` |
| GitLab PAT | `\bglpat-[A-Za-z0-9_-]{20,}` |
| Atlassian API token | `\bATATT[A-Za-z0-9_=-]{20,}` |
| Anthropic API key | `\bsk-ant-[A-Za-z0-9_-]{20,}` |
| OpenAI API key | `\bsk-[A-Za-z0-9]{40,}` |
| Stripe live secret | `\bsk_live_[A-Za-z0-9]{20,}` |
| Stripe restricted | `\brk_live_[A-Za-z0-9]{20,}` |
| AWS access key id | `\bAKIA[0-9A-Z]{16}\b` |
| AWS session key id | `\bASIA[0-9A-Z]{16}\b` |
| JWT | `\beyJ[A-Za-z0-9_-]{10,}\.[A-Za-z0-9_-]{10,}\.[A-Za-z0-9_-]{10,}` |
| Private key block | `-----BEGIN (?:RSA |EC |OPENSSH |DSA )?PRIVATE KEY-----` |

(Yes, that's 15 patterns — "Private key block" is the bonus 15th. The
header comment in the script says "14 known credential prefixes" because
it counts the BEGIN-PRIVATE-KEY pattern as a separate non-prefix class.)

## Heuristic detectors

In addition to the prefix patterns:

- **AWS secret-key shape** — 40 chars in `[A-Za-z0-9/+]`. AWS secret
  access keys don't have a recognizable prefix, so this is the next-best
  heuristic.
- **High-entropy hex** — any hex string ≥ 32 chars. Catches arbitrary
  API tokens that lack a recognizable prefix.

The high-entropy detector uses a Shannon-bits-per-symbol calculation to
filter out cases where the hex-looking string is actually low-entropy
(e.g. a `0` chain or a repeating pattern).

```js
function shannonBits(s) {
  const freq = {};
  for (const c of s) freq[c] = (freq[c] || 0) + 1;
  let bits = 0;
  for (const c of Object.keys(freq)) {
    const p = freq[c] / s.length;
    bits -= p * Math.log2(p);
  }
  return bits;
}
```

The threshold is calibrated to flag actual tokens while allowing
hex-looking strings that are clearly not secrets (commit SHAs are short
enough to fall under 32 chars; UUIDs hit the high-entropy filter but are
typically formatted with dashes that break the `\b[a-f0-9]{32,}\b`
match).

## How it walks the payload

The hook walks every string in `tool_input`, including deeply nested
objects and arrays, and reports the JSON-path of the offending field
back to the agent:

```js
function* walkStrings(value, path = "") {
  if (typeof value === "string") {
    yield { path, value };
  } else if (Array.isArray(value)) {
    for (let i = 0; i < value.length; i++)
      yield* walkStrings(value[i], `${path}[${i}]`);
  } else if (value && typeof value === "object") {
    for (const [k, v] of Object.entries(value))
      yield* walkStrings(v, path ? `${path}.${k}` : k);
  }
}
```

So if a token shows up inside `tool_input.attachments[0].text`, the
denial message names exactly that path. The agent can fix the specific
field rather than re-emit the entire payload.

## The denial output

```text
✗ Secret(s) detected in tool input — denying call.

Hits:
  - body: Anthropic API key
  - blocks[0].text: GitHub PAT

Remove the secret(s) and retry. Credentials must never be sent through
Slack/Jira write tools — use a secret-rotation flow instead.
```

`permissionDecision: "deny"` is a **hard block** — unlike `"ask"`, the
user is not prompted to override it. The agent sees the denial and adjusts.

<Aside type="caution">
The scanner runs **before** the tool call leaves the agent's context, but
**after** the model has already seen the secret (because the model
constructed the tool input). The first line of defense against secrets
ending up in a model context is *not putting them there in the first
place*: keep them in environment variables, scoped at the MCP server
layer, never echoed back to the model via prompts or tool descriptions.
</Aside>

## False positives

These patterns are tuned to be precise. The two heuristics (AWS-shape
and high-entropy hex) can occasionally fire on legitimate strings —
specifically:

- Commit SHAs ≥ 32 chars (e.g. SHA-256 hashes). Rare in chat / ticket
  bodies, but happens.
- Base64-encoded binary content like images or compressed payloads.
  These shouldn't typically appear in Slack message text or Jira
  comments — if they do, something else is wrong.

When the hook denies a legitimate string, the agent can rephrase
(truncate, hyphenate, paraphrase) and retry. The "deny" is bounded — it
doesn't kill the session, it just blocks the current tool call.

## Tests

`tests/hooks.test.ts` covers the scanner with:

- One positive case per credential pattern.
- Clean strings that look superficially secret-shaped but aren't (commit
  SHAs, UUIDs with dashes).
- Nested payload walking — a token deep inside an array of objects must
  still be found.
- Multiple hits in one payload — the denial reports all of them, not
  just the first.
